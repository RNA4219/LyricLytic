# LyricLytic 韻ガイド要件 v1

## 1. 目的

本書は、LyricLytic における `韻ガイド / 母音・子音・モーラ表示` の要件を定義する。  
対象は、歌詞入力中に「発音の近さ」「末尾の響き」「韻の揃い方」を確認しやすくする補助機能である。

本機能は自動採点ではなく、作詞時の判断材料を増やすための補助とする。

## 2. 基本方針

- 韻候補検索の主軸は `辞書 + 音韻正規化 + インデックス` で行う
- LLM は意味拡張、言い換え、再ランキング、自然さ補正のみに使う
- 韻候補検索そのものを LLM に丸投げしない
- UI は編集画面の文脈を壊さず、歌詞入力欄の補助ビューとして表示する

## 3. 辞書層の採用方針

### 3.1 実行時主辞書

- `SudachiDict-core` を実行時の主辞書とする
- 実装系は `SudachiPy` または `sudachi.rs` のどちらでもよい
- 将来の拡張に備え、`small / core / full` を切り替え可能な設計にする

採用理由:

- 速度、保守性、実運用のバランスがよい
- 読み取得と分かち書きの品質が安定している
- 実装切替の選択肢がある

### 3.2 新語補完

- `NEologd` を新語、固有名詞、ネット語、作品名の補完に使う
- 実行時に常時マージしてもよいし、辞書更新ジョブで索引化してもよい

### 3.3 検証系辞書

- `UniDic` は実行時の第一候補ではなく、検証・差分監査・正規化ルール確認の参照系として扱う
- 実行時依存を増やしすぎないことを優先する

### 3.4 ライセンス運用

- `UniDic` を配布物に含める場合は、配布ライセンス表記、AUTHORS、NOTICE を事前に整理する
- ライセンス義務を実装開始前に `docs/licenses/` または同等の場所へ明文化する

## 4. 音韻解析パイプライン

実装の既定フローは以下とする。

1. Sudachi で分かち書きと読みを取得
2. 読みを `かな / モーラ列 / 母音列 / 子音列 / ローマ字列` に正規化
3. 末尾 `n` モーラ一致、末尾母音一致、子音母音パターン一致で候補抽出
4. NEologd 由来語を候補へ補完
5. UniDic でズレや誤判定を検証
6. 必要時のみ LLM で意味補正や自然さ補正を行う

## 5. 役割分担

### 5.1 辞書 / ルールベースで行うこと

- 分かち書き
- 読み取得
- かな正規化
- モーラ列生成
- 母音列生成
- 子音列生成
- 末尾一致 / パターン一致による候補抽出
- 韻の強弱スコアリング

### 5.2 LLM で行うこと

- 候補の意味的な言い換え
- 文脈に合う順への再ランキング
- ぴったり韻がない時の近似案生成
- 歌詞全体の自然さ補正

### 5.3 LLM で行わないこと

- 最初の韻候補検索
- 読み取得の主処理
- 音韻一致判定の主処理

## 6. UI 要件

### 6.1 編集画面での表示

- 韻ガイドは編集画面下部ペインで表示する
- 行ごとに `原文 / ローマ字 / 母音列 / 子音列` を確認できる
- セクションタグ行 `[Intro]` などは文字数カウントと同様にガイド対象外としてよい
- `ALL` と個別セクションのどちらでも動作する

### 6.2 将来の表示拡張

- 末尾一致色付け
- 候補行間の近似強調
- 韻候補クリックで本文へコピー
- 類韻候補一覧パネル

## 7. PoC / MVP の段階適用

### 7.1 PoC で必須

- 音韻ガイド機能を正式機能として持つ
- フロント側で `原文 / ローマ字 / 母音列 / 子音列` を確認できる
- 実装境界を `辞書層 / 正規化層 / LLM補正層` に分ける
- 現行フォールバック実装があっても、将来の Sudachi 差し替えに耐える構造にする

### 7.2 PoC で許容

- Sudachi 本体が未同梱の場合、フォールバック正規化ロジックで暫定稼働してよい
- ただし、フォールバックは正本実装ではなく暫定と明示する

### 7.3 MVP 以降で必須化を検討

- SudachiDict-core 常時利用
- NEologd 補完の正式組み込み
- UniDic との差分検証ジョブ
- 韻候補抽出 UI
- LLM 補正の再ランキング UI

## 8. 実装制約

- 辞書層は UI から直接参照させず、`rhyme analysis service` を介す
- フロントは `表示` に集中し、読み取得や辞書結合の責務を持たない
- Sudachi を Rust 側へ寄せる場合も、フロントとの契約は `RhymeGuideRow[]` のような表示用 DTO へ変換する

## 9. 受け入れ観点

- 日本語歌詞に対して `ローマ字 / 母音列 / 子音列` が表示される
- かな / カナ / 一部英字混在でも最低限破綻しない
- セクションタグ行はガイド対象外になる
- フォールバック実装でも UI 契約が変わらない
- Sudachi 差し替え時に Editor 側の描画コードを大きく変えずに済む

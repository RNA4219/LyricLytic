# LyricLytic フロントエンド要件定義 v1

## 1. 目的

本書は、`requirements.md`、`screen-flow-v1.md`、`acceptance-test-cases-v1.md` をもとに、LyricLytic のフロントエンド実装に必要な要件だけを再編した文書である。  
目的は、UI 実装担当者が「どの画面に、何を、どの挙動で、どの優先順位で置くか」を判断しやすくすることにある。

本書は新規要件を追加するものではなく、既存要件のフロントエンド解釈を明確にするための整理文書とする。

## 2. 対象範囲

本書の対象は以下とする。

- 画面構成
- 画面内レイアウト
- UI コンポーネント責務
- 主要操作導線
- 状態表示
- エラー表示
- OS ごとの UI 差分
- フロントエンド側の受け入れ観点

本書の対象外は以下とする。

- SQLite スキーマの詳細
- Rust 側コマンド実装
- LLM API 通信処理の内部実装
- Tauri capability や CSP の具体設定値

## 3. フロントエンド基本方針

### 3.1 編集コンテキスト維持

創作中の集中を壊さないことを最優先とする。  
検索、StyleProfile 編集、断片参照、コピー設定確認などは、原則として編集画面から別画面へ強く遷移させず、パネル、ドロワー、モーダルで完結させる。

### 3.2 Working Draft 主体

フロントエンドは、通常の編集対象が `Working Draft` であることを明確に表示しなければならない。  
`LyricVersion` は参照、比較、復元起点として扱い、直接編集対象として誤認させない。

### 3.3 非破壊的な操作理解

保存は `履歴版を増やす操作`、復元は `Working Draft 再構築`、削除は `論理削除` であることを UI 上で誤解なく伝える必要がある。

### 3.4 ローカル完結の可視化

外部通信やクラウド同期が前提ではないことを UI が壊さないようにする。  
外部 URL は参照情報として扱い、明示操作なしに開かない。

## 4. 画面一覧

MVP / PoC の主要画面は以下とする。

1. ホーム / プロジェクト選択画面
2. 歌詞編集ワークスペース
3. 差分確認ビュー
4. 削除済みデータ管理画面
5. 保存ダイアログ
6. インポートダイアログ

検索、曲紐付け、断片、StyleProfile は専用画面を持たず、歌詞編集ワークスペース内のパネルまたはインスペクタで扱う。

## 5. 画面別要件

### 5.1 ホーム / プロジェクト選択画面

役割:

- 起動直後の入口
- Project 一覧表示
- 新規 Project 作成
- 既存 Project 再開

必須 UI:

- Project 一覧
- 新規 Project 作成ボタン
- 空状態表示
- 最終更新日時の視認

挙動要件:

- 新規 Project 作成後は歌詞編集画面へ遷移する
- 削除済み Project は通常一覧に表示しない
- 一覧順は最終更新順を基本とする

### 5.2 歌詞編集ワークスペース

役割:

- 日常的な主作業画面
- Working Draft 編集
- セクション編集
- 断片参照
- 検索
- StyleProfile 編集導線
- 保存、コピー、AI 補助起点

必須レイアウト:

- 左ペイン: Project 一覧、LyricVersion ツリー、Section 一覧
- 中央ペイン: Monaco Editor による本文 / セクション編集
- 右ペイン: メタ情報、コピー、保存、断片、推敲メモ、StyleProfile、曲紐付け、検索導線

必須表示:

- 現在の編集対象が Working Draft であること
- 直近保存済み LyricVersion 名または履歴への導線
- 自動保存状態
- AI 接続状態への導線

挙動要件:

- 空 Draft から開始できる
- セクションプリセットは追加候補として提示し、自動挿入しない
- 検索 UI と StyleProfile 編集 UI は編集画面から離脱せず開ける
- 曲紐付け、断片一覧、履歴参照も可能な限り同一画面内で扱う
- 断片挿入後も編集コンテキストを維持する
- 左ナビゲーション / 中央主作業 / 右インスペクタの 3 ペインを基本とする

### 5.3 差分確認ビュー

役割:

- LyricVersion 同士の比較
- 復元起点の確認

必須レイアウト:

- 左ペイン: 比較対象一覧
- 中央ペイン: Monaco Diff Editor
- 右ペイン: 差分サマリー、変更件数、復元導線

挙動要件:

- 比較対象 2 版を選べる
- 少なくとも行単位の差分が分かる
- `この版から作業を再開` の導線を持つ
- 完全な別世界へ遷移させず、フルスクリーンモーダル、タブ、またはエディタ置換ビューとして開いてよい

### 5.4 曲紐付けインスペクタ

役割:

- SongArtifact 登録と参照

必須レイアウト:

- 歌詞編集ワークスペース右ペイン内の詳細インスペクタとして配置する
- URL / ファイル登録 UI
- 曲詳細、評価メモ、プロンプトメモ、スタイルメモ

挙動要件:

- 紐付け対象の LyricVersion を明示する
- Working Draft は紐付け対象に出さない
- URL オープン時は確認ダイアログを経由する

### 5.5 削除済みデータ管理画面

役割:

- 論理削除済みデータの確認
- バッチ単位復元

必須 UI:

- 削除バッチ一覧
- 種別フィルタ
- バッチ内個別対象の展開
- 復元ボタン

対象データ:

- Project
- LyricVersion
- SongArtifact
- CollectedFragment
- RevisionNote
- StyleProfile

挙動要件:

- 物理削除 UI は通常導線に出さない
- Project / LyricVersion はバッチ単位で復元する
- 復元不可時は理由を表示する
- 主作業導線ではなく、Settings 配下、More 配下、または Project 詳細のサブ階層に置く

### 5.6 保存ダイアログ

役割:

- LyricVersion 保存時のメタ入力

必須 UI:

- `snapshotName` 入力
- 任意メモ入力
- 保存対象説明

挙動要件:

- 保存後も編集対象は Working Draft のまま
- 保存は履歴版追加であり、編集モード切替ではないことを説明する

### 5.7 インポートダイアログ

役割:

- `.txt` から断片または本文を取り込む

必須 UI:

- ファイル選択
- 断片取り込み / 本文取り込みの選択
- 文字コード失敗時の再選択導線

挙動要件:

- Windows / macOS とも OS ネイティブダイアログを使う
- PoC では `.txt` 単一ファイル選択を既定とする

## 6. 共通 UI 要件

### 6.1 検索 UI

- 専用画面は持たない
- 編集画面から開くパネル、ドロワー、またはモーダルとする
- `本文 / 過去版 / 断片 / タグ` の種別を 1 つの UI で切り替える
- 過去版横断検索に RevisionNote は含めない
- 検索結果選択後も編集コンテキストを維持する

### 6.2 エラー表示

- 保存失敗、読込失敗、コピー失敗、インポート失敗、LLM 接続失敗をユーザーが区別できる文言を出す
- エラー表示は創作作業を不必要に遮らない
- 再試行可能な処理は再試行導線を持つ

### 6.3 進行中表示

- 保存中
- 差分読込中
- インポート中
- エクスポート中
- LLM 実行中

上記は、処理中であることを視認できる状態表示を持つこと。

### 6.4 確認ダイアログ

必須対象:

- Project 削除
- LyricVersion 削除
- Working Draft を置き換える復元
- 外部 URL オープン

### 6.5 ショートカット表示

- Windows では `Ctrl`
- macOS では `Command`

PoC で最低限対象にする操作:

- 保存
- 検索
- コピー
- 差分確認
- 曲紐付け
- インポート
- エクスポート
- 削除済みデータ管理

## 7. UI 状態管理の前提

フロントエンドは少なくとも以下の状態を扱える必要がある。

- アクティブ Project
- アクティブ Working Draft
- 選択中 Section
- LyricVersion 一覧
- 差分比較対象
- CollectedFragment 一覧
- SongArtifact 一覧
- StyleProfile
- 検索 UI の開閉状態
- 保存ダイアログの開閉状態
- インポートダイアログの開閉状態
- 削除済みデータ管理画面のフィルタ状態
- LLM 接続状態
- グローバル通知状態

状態管理上の制約:

- Working Draft と LyricVersion を同一編集状態として扱わない
- `app_settings` と Project 固有状態を混同しない
- 検索 UI や右ペイン補助 UI は、編集中の本文状態を破壊しない

## 8. コンポーネント責務

### 8.1 ページレベル

- `HomePage`
- `EditorPage`
- `DiffPage`
- `DeletedItemsPage`

### 8.2 主要パネル

- ProjectListPanel
- VersionTreePanel
- SectionListPanel
- EditorPanel
- SearchPanel
- FragmentPanel
- RevisionNotePanel
- StyleProfilePanel
- SongLinkInspector
- CopyOptionsPanel
- SaveDialog
- ImportDialog

### 8.3 補助コンポーネント

- StatusBadge
- LoadingIndicator
- ErrorBanner
- ConfirmDialog
- ShortcutHint

責務原則:

- 画面責務とドメイン状態責務を分離しやすい構成にする
- Monaco / Diff Editor のラッパーは UI 側で閉じ込める
- 外部 URL オープンやファイルダイアログ起動などの副作用は薄い呼び出し境界に寄せる

## 9. OS 差分要件

フロントエンドで意識すべき OS 差分は以下に限定する。

- ショートカット表記
- OS ネイティブファイルダイアログ利用
- パス表示
- 外部 URL オープン確認

フロントエンドで保証しなくてよいこと:

- SongArtifact ローカルパスの OS 間可搬性
- 保存先を OS ネイティブに開く最適化
- macOS 専用メニューバー最適化
- Windows 専用コンテキストメニュー最適化

## 10. 非機能要件

### 10.1 操作性

- キーボード中心で操作できること
- 補助 UI が本文編集を邪魔しないこと
- 検索や右ペイン操作で編集対象を見失わないこと

### 10.2 応答性

- 通常の本文編集は 100ms 未満で応答することを目標とする
- Project 切替と履歴一覧表示は 1 秒以内を目標とする
- 全文コピーは 1 秒以内に完了すること

### 10.3 可逆性

- 復元は Working Draft 再構築として扱い、既存 LyricVersion を破壊しない
- 削除は論理削除を前提とし、削除済みデータ管理画面から復元できること

## 11. フロントエンド受け入れ観点

以下はフロントエンド観点で最低限合格が必要なケースである。

- 新規 Project 作成後に空 Draft へ遷移できる
- セクション追加と連番付与が分かる
- 自動保存状態が破綻なく表示される
- 保存ダイアログから LyricVersion を作成できる
- 差分確認画面で 2 版比較できる
- 検索パネルが編集コンテキストを維持して動作する
- 断片を挿入できる
- StyleProfile を編集できる
- 削除済みデータ管理画面から復元できる
- 外部 URL オープン確認ダイアログが表示される
- Windows / macOS でショートカット表記が切り替わる

詳細テストケースは `acceptance-test-cases-v1.md` を正本とする。

# LyricLytic 要件定義

## 1. 文書の目的

本書は、AI 音楽生成サービス向けの歌詞制作を支援するデスクトップアプリケーション「LyricLytic」の要件を定義するものである。  
本プロダクトは、歌詞の断片収集、構成整理、推敲、バージョン管理、曲との紐付け、外部サービス向けコピーを一貫して扱う環境を提供する。

本書では、既存の構想資料と詳細要件テキストを統合し、MVP で実装する範囲、ドメインモデル、機能要件、非機能要件、セキュリティ方針、受け入れ基準を整理する。

### 1.1 文書ステータス

本書は、実装判断の基準として使える状態まで段階的に引き上げる対象とする。  
今後のレビューでは、曖昧さを減らし、実装者が判断に迷う余地を減らすことを重視する。

### 1.2 この文書のゲート条件

本書は、少なくとも以下を満たした時点で「実装着手可能」と判断する。

- MVP 範囲と非対象範囲が衝突なく定義されていること
- 主要エンティティの責務と関係が定義されていること
- 主要操作の正常系が受け入れ基準に落ちていること
- データ保存、削除、復元の扱いが定義されていること
- 実装上の前提と未確定事項が区別されていること
- 非機能要件に最低限の判定基準があること

## 2. プロダクト概要

LyricLytic は、Suno や Udio などの AI 音楽生成サービスを使って楽曲制作を行う個人クリエーター向けの、歌詞制作専用デスクトップワークベンチである。  
歌詞全文を自動生成するツールではなく、歌詞を編集・推敲・比較・管理するツールとして位置付ける。

フロントエンドの本文編集エンジンには Monaco Editor を採用し、Tauri をラッパーとしてデスクトップアプリ化する。  
アプリは完全ローカル動作を前提とし、歌詞データ、履歴、補助判定、曲紐付け情報はローカル環境内で完結する。

## 3. 背景と解決したい課題

対象ユーザーは、AI 音楽生成サービスへ歌詞を何度も貼り付け、試行錯誤しながら楽曲を育てる。現状では、以下のような課題がある。

- 歌詞の断片がメモ、テキスト、脳内に散らばり整理しづらい
- 外部エディタと生成サービスの往復が多く、コピー導線が弱い
- 改稿履歴や良かった旧版を見失いやすい
- どの歌詞版からどの曲を生成したか追跡しづらい
- 推敲時に「ありがちな表現」や言い換え候補を確認する補助が不足している

LyricLytic はこれらを解消し、歌詞制作の思考を止めないワークフローを提供する。

## 4. 対象ユーザー

本プロダクトの対象ユーザーは以下を想定する。

- AI 音楽生成サービスを用いて楽曲制作を行う個人クリエーター
- 歌詞を外部エディタやメモで試行錯誤しているユーザー
- 作詞案の差分、旧版、採用候補を整理したいユーザー
- 全文自動生成よりも推敲支援を重視するユーザー

## 5. 基本方針

### 5.1 ローカル完結

本プロダクトは完全ローカルで動作する。  
歌詞本文、断片、履歴、曲紐付け情報、推敲メモ、補助判定結果はすべてローカル保存する。

### 5.2 編集・推敲中心

本プロダクトは編集・推敲・比較・管理を中核とする。  
AI 支援は局所的なレビューや候補提示に限定し、全文ドラフト生成を主機能とはしない。

### 5.3 創作の邪魔をしない

補助機能は、創作を矯正したり強制したりするものではなく、判断材料を増やすための補助輪として設計する。  
警告やレコメンドはすべて参考情報であり、品質の真理値を返すものではない。

### 5.4 歌詞特有の構造を扱う

自由文エディタではなく、セクション構造、反復、断片流し込み、推敲メモ、履歴比較など、歌詞制作に特有の操作を支援する。

## 6. スコープ

### 6.1 MVP に含めるもの

MVP では、歌詞制作ワークフローの骨格と詳細要件を一体で扱う。少なくとも以下を含む。

- プロジェクト管理
- 歌詞編集
- セクション管理
- 自動保存
- スナップショット保存
- バージョン履歴
- 差分比較
- 推敲メモ
- 断片管理
- 断片から本文への挿入
- 曲紐付け
- 全文 / セクション単位コピー
- 検索
- モデル頻出表現チェック
- 低頻出候補レコメンド
- 韻ガイド表示
- Style 補助情報保持
- テキストインポート
- エラー表示

### 6.2 MVP に含めないもの

以下は MVP の対象外とする。

- 外部 API 連携
- クラウド同期
- モバイルアプリ
- 共同編集
- 本プロダクト内での全文ドラフト自動生成
- 著作権上不明瞭な歌詞データを前提にした学習機能
- 高度な自動採点型 NLP 解析や本格的な韻品質スコアリング
- テーマ切り替えや大規模カスタマイズ機能
- 同時編集競合解決

## 7. 想定ワークフロー

本プロダクトが支援する主要ワークフローは以下の通り。

1. Collect: 断片やフレーズを収集する
2. Structure: 断片を並べ、歌詞構成を組み立てる
3. Revise: 声に出しながら推敲し、表現を磨く
4. Link Song: 生成した曲と歌詞版を紐付ける
5. Copy: 外部サービスへ歌詞をコピーする
6. History: 過去版へ戻る、比較する

```mermaid
flowchart LR
    A[Collect] --> B[Structure]
    B --> C[Revise]
    C --> D[Link Song]
    D --> E[Copy]
    E --> F[History]
```

## 8. 前提・制約

### 8.1 利用環境前提

- 単一ユーザーによるローカル利用を前提とする
- 初期リリース時点ではデスクトップ利用のみを想定する
- 常時オンライン接続を前提にしない

### 8.2 著作物・参照情報の扱い

- 歌詞本文はユーザーの創作物としてローカル保存する
- 曲生成サービスの利用条件確認はユーザー責任とし、アプリは法的判断を代替しない
- 外部サービス由来の URL やメタ情報は参照用データとして保持する

### 8.3 実装判断の既定値

- 未確定事項が解消されるまでは、データ保全を優先する
- 削除操作は即時物理削除ではなく、復元可能な扱いを優先する
- AI 補助はすべて明示実行とし、自動実行を既定にしない
- 初期対応 OS は Windows と macOS を必須対象とし、Linux は将来対応候補とする
- 断片インポート形式は MVP では UTF-8 の `.txt` を必須対象とする
- バックアップは MVP では自動同期ではなく、手動エクスポートを優先する
- PoC の優先検証対象は Windows 11 と macOS 最新安定版とする

## 9. 用語定義

### 9.1 Project

歌詞制作の単位。テーマ、タグ、メモ、歌詞履歴、断片、曲紐付け情報を包含する。

### 9.2 LyricVersion

歌詞のスナップショット単位。明示保存された版であり、履歴比較・復元の対象となる。

### 9.3 Working Draft

編集中の最新状態。自動保存されるが、必ずしも正式な履歴版ではない。

### 9.4 Section

歌詞を構成する論理単位。例として Intro / Verse / Pre-Chorus / Chorus / Bridge / Outro を持つが、任意追加も可能とする。

### 9.5 RevisionNote

推敲メモ。特定の版・特定のセクションに対して、弱さ、フック候補、言い換え候補、保留事項などを記録する。

### 9.6 StyleProfile

MVP では Project に対して付与する補助的スタイル情報。曲調や歌詞の方向性を保持するための情報であり、品質判定の真値ではない。  
LyricVersion 単位または候補案単位への付与は将来拡張とする。

### 9.7 モデル頻出表現チェック

ローカル LLM に複数回生成を行わせ、指定セクションに対して出やすい表現を割合付きで可視化する機能。創作上の良否判定ではなく、モデル偏りに基づくレビュー補助として扱う。

### 9.8 低頻出候補レコメンド

モデル頻出表現チェックの結果を踏まえ、相対的に出現割合の低い別案候補を提示する機能。品質保証ではなく、局所リライト支援として扱う。

## 10. ドメインモデル

### 10.1 Project

Project は以下の情報を持つ。

- projectId
- title
- theme
- tags
- memo
- styleProfile
- createdAt
- updatedAt

### 10.2 LyricVersion

LyricVersion は以下の情報を持つ。

- lyricVersionId
- projectId
- snapshotName
- bodyText
- parentLyricVersionId
- createdAt
- note
- copyIncludeHeadings
- copyPreserveBlankLines

LyricVersion は明示保存されたスナップショットである。  
一覧表示用のタイトルは `snapshotName` を基準とし、必要に応じて作成日時等から導出してよい。

### 10.3 Working Draft

Working Draft は以下の情報を持つ。

- workingDraftId
- projectId
- latestBodyText
- latestSectionState
- updatedAt

Working Draft は自動保存対象であり、履歴版とは区別される。
PoC では `draft_sections` を正本とし、`latestBodyText` は全文プレビューおよびコピー用の派生キャッシュとして扱う。

### 10.4 Section

Section は以下の情報を持つ。

- sectionId
- lyricVersionId または workingDraftId
- sectionType
- displayName
- order
- bodyText
- note

Section には表示名とは別に内部一意識別子 SectionID を必須とする。  
表示名の重複が発生しうるため、内部処理は表示名ではなく SectionID を基準に行う。

Section は論理概念であり、永続化実装では所有先が `LyricVersion` か `Working Draft` かを曖昧にしない構造を採ること。

### 10.5 RevisionNote

RevisionNote は以下の情報を持つ。

- revisionNoteId
- lyricVersionId
- sectionId
- rangeStart（任意）
- rangeEnd（任意）
- noteType
- comment
- createdAt

RevisionNote は LyricVersion + Section を主参照先とする。  
文字範囲は補助情報であり、絶対参照としては扱わない。
RevisionNote が参照する Section は、同一 LyricVersion に属するセクションでなければならない。

### 10.6 SongArtifact

SongArtifact は以下の情報を持つ。

- songArtifactId
- projectId
- lyricVersionId
- serviceName
- songTitle
- url または filePath
- promptMemo
- styleMemo
- evaluationMemo
- createdAt

### 10.7 CollectedFragment

CollectedFragment は以下の情報を持つ。

- collectedFragmentId
- projectId
- text
- source
- tags
- status
- createdAt

status は少なくとも `unused` / `used` / `hold` を持てること。

### 10.8 StyleProfile

StyleProfile は以下の情報を持つ。

- styleProfileId
- projectId
- tone
- vocabularyBias
- tabooWords
- structurePreference
- tags
- memo

StyleProfile は思考補助・検索補助のための情報であり、品質判定の根拠としては扱わない。

## 11. エンティティ関係と制約

### 11.1 関係

- 1 Project は 0 個以上の LyricVersion を持てること
- 1 Project は常に 0 または 1 つの Working Draft を持つこと
- 1 Project は 0 個以上の SongArtifact を持てること
- 1 Project は 0 個以上の CollectedFragment を持てること
- 1 Project は 0 または 1 つの active StyleProfile を持てること
- 1 LyricVersion は 0 個以上の Section を持てること
- 1 Working Draft は 0 個以上の Section を持てること
- 1 LyricVersion は 0 個以上の RevisionNote を持てること
- 1 LyricVersion は 0 個以上の SongArtifact から参照されうること

### 11.2 制約

- 1 SongArtifact は必ず 1 つの Project と 1 つの LyricVersion に属すること
- 1 RevisionNote は必ず 1 つの LyricVersion と 1 つの Section を参照すること
- Section は LyricVersion または Working Draft のいずれか一方にのみ属すること
- Working Draft は Project ごとに 1 件を上限とすること
- StyleProfile は Project ごとに active 1 件を上限とすること
- LyricVersion の親参照は同一 Project 内に限定すること
- 論理削除を採るため、一意性制約は「有効データに対してのみ成立する」前提で設計すること
- 同一 Project 内参照など、RDB の単純外部キーだけで表現しづらい制約はアプリケーション層でも担保すること

## 12. データライフサイクル要件

### 12.1 作成

- Project 作成時に空の Working Draft を持てること
- LyricVersion は保存操作により新規作成されること
- SongArtifact と CollectedFragment は Project 配下に作成されること

### 12.2 更新

- Working Draft は自動保存で上書き更新されること
- PoC では Section 単位の更新結果から `latestBodyText` を再構成すること
- LyricVersion は不変オブジェクトとして扱い、直接上書きしないこと
- 過去版を元に再編集する場合は、選択した LyricVersion の内容で Working Draft を再構築すること

### 12.3 削除

- Project 削除時は、配下の LyricVersion、Working Draft、SongArtifact、CollectedFragment、RevisionNote をまとめて削除対象として扱うこと
- Project 削除時は、配下の StyleProfile とそのタグも削除対象として扱うこと
- MVP では誤削除防止のため、Project 削除前に確認ダイアログを必須とすること
- LyricVersion 単体削除は MVP では確認付きの明示操作に限定すること
- RevisionNote、CollectedFragment、SongArtifact の削除は復元可能な運用を優先すること
- MVP の削除方式は論理削除を既定とし、物理削除はメンテナンス用途または将来機能とすること
- 同一削除操作で削除されたデータ群は `deleted_batch_id` で束ねること
- Project 削除時は Project 配下の論理削除対象を同一 `deleted_batch_id` でまとめて削除すること
- LyricVersion 削除時は対象 Version、VersionSection、RevisionNote を同一 `deleted_batch_id` でまとめて削除すること
- LyricVersion 削除時に SongArtifact は既定で連鎖削除しないこと

### 12.4 復元

- 自動保存された Working Draft から前回終了時の作業を再開できること
- LyricVersion は履歴から任意時点を選んで復元開始できること
- 復元は既存版の破壊的上書きではなく、選択した版の内容で Working Draft を再構築する挙動とすること
- 論理削除された SongArtifact、CollectedFragment、RevisionNote、StyleProfile は UI 上から復元できること
- Project と LyricVersion の復元は `deleted_batch_id` 単位で扱えること
- Project 復元時に同一 Project がすでに active 状態で存在する場合、MVP では復元不可とすること

### 12.5 エクスポート / バックアップ

- MVP では Project 単位での手動エクスポート機能を持つこと
- 自動クラウドバックアップは MVP 対象外とする
- エクスポート形式は可搬性の高い構造化形式を優先し、MVP では zip パッケージ内の UTF-8 JSON を正本とする
- エクスポートには人間確認用のテキスト出力を補助的に同梱できること
- Project export の対象は当該 Project 配下データに限定し、app_settings は含めないこと
- 論理削除データは既定ではエクスポート対象に含めず、必要時のみ明示オプションで含められること
- SongArtifact が参照するローカルファイルの実体コピーは MVP のエクスポート対象外とし、パス文字列のみ保持すること

## 13. セクション要件

### 13.1 セクションプリセット

歌詞編集画面では、以下のセクションプリセットを追加候補として提供する。  
新規 Project 作成直後の Working Draft は空状態を許容し、プリセットを自動挿入しない。

- Intro
- Verse
- Pre-Chorus
- Chorus
- Bridge
- Outro

### 13.2 セクション追加

ユーザーは任意のセクションを追加できること。  
既存プリセットに縛られず、自由に名称を定義できること。

### 13.3 セクション命名規則

同名セクションが既に存在する場合、新規追加時には末尾に連番を付与する。

例:

- Verse
- Verse1
- Verse2

この連番付与は追加時のみ実施する。  
常時監視による再命名や、既存名の自動再計算は行わない。

### 13.4 セクション識別

セクションは表示名とは別に SectionID を持つ。  
メモ、差分、内部処理では SectionID を基準に扱う。

### 13.5 セクション編集

1 つの歌詞編集画面の中で、複数セクションを連続的に編集できること。  
セクション構造は補助的な論理単位であり、楽曲構造を強制しない。
PoC ではセクション配列を編集の正本とし、全文表示はセクション連結結果として扱うこと。

## 14. 保存・履歴要件

### 14.1 自動保存

編集中の状態は自動保存されること。  
自動保存された状態は作業復帰用の最新状態として扱う。

### 14.2 スナップショット保存

保存ボタン押下時、その時点の内容を LyricVersion としてスナップショット保存すること。  
これが正式な履歴版となる。

保存後も通常の編集対象は Working Draft のままとし、保存操作自体は履歴版を増やす操作として扱うこと。

### 14.3 履歴管理

スナップショット版は履歴一覧から参照できること。  
任意の版を選択し、比較・復元できること。

過去版からの復元は、既存 LyricVersion を上書きするのではなく、選択した版の内容で Working Draft を再構築する操作として扱うこと。

### 14.4 自動保存状態と履歴版の区別

Working Draft と LyricVersion は明確に区別すること。  
自動保存は事故防止のため、履歴版は意図ある保存のために存在する。

### 14.5 保存時メタ情報

- LyricVersion 保存時に snapshotName を任意入力できること
- snapshotName 未入力時は作成日時ベースの既定名を付与できること
- 保存時に任意メモを残せること

### 14.6 競合前提

- 単一端末・単一ユーザー前提のため、同時編集競合の解決機構は MVP 範囲外とする
- ただしアプリ異常終了後でも、直近の自動保存状態が破損しにくい保存方式を採ること

### 14.7 保持期間

- 論理削除データの保持期間は MVP では無期限を既定とする
- 将来、物理削除ポリシーを導入する場合は別要件として定義する

## 15. 機能要件

### 15.1 プロジェクト管理

- プロジェクトの作成、選択、編集、削除ができること
- タイトル、テーマ、タグ、メモ、Style 情報を保持できること
- 起動時に表示するホーム / プロジェクト選択導線を持つこと

### 15.2 歌詞編集

- Monaco Editor を用いて歌詞本文を編集できること
- セクション単位での編集を支援できること
- キーボード中心で快適に操作できること

### 15.3 バージョン管理

- 保存ボタンでスナップショットを作成できること
- 親子関係を保持できること
- 差分比較ができること
- 任意の版へ戻れること
- 任意の版から Working Draft を再構築できること

### 15.4 差分比較

- 2 つの版の差分を確認できること
- 少なくとも行単位での変更が分かること
- Diff Editor を用いて視覚的に差分を把握できること

### 15.5 推敲メモ

- 版 + セクションに対して推敲メモを付与できること
- メモ種別を設定できること
- 必要に応じて範囲指定できること
- メモは他版へ自動継承しないこと

### 15.6 断片管理

- 断片を登録できること
- 断片を一覧・検索できること
- 断片を本文へ挿入できること
- 断片を使用済み、保留などに分類できること
- 断片の出所を自由記述で記録できること
- 断片を複数プロジェクトで共有しないことを MVP の既定とすること

### 15.7 検索

- 本文内検索ができること
- 過去版横断検索ができること
- 断片検索ができること
- タグ検索ができること
- MVP の本文内検索対象は、現在開いている Working Draft の本文および現在表示中の Section とすること
- MVP の過去版横断検索対象は LyricVersion 本文および version_sections 本文とし、RevisionNote は含めないこと
- MVP の断片検索対象は CollectedFragment の `text` と `source` とすること
- MVP のタグ検索対象は Project、CollectedFragment、StyleProfile に付与されたタグとすること
- PoC ではタグ検索対象を CollectedFragment のタグのみに縮退し、Project/StyleProfile タグは PoC 範囲外とすること
- MVP の検索 UI は専用画面ではなく、歌詞編集画面から開く検索パネルまたは同等の非遷移 UI とすること
- PoC では `本文 / 過去版 / 断片 / タグ` の検索種別を同一検索 UI 内で切り替えられること

### 15.8 コピー

- 全文コピーができること
- セクション単位コピーができること
- 見出し込み / 見出しなしを選べること
- 空行保持 / 詰めるを選べること
- 最後に使った整形設定を保持できること
- コピー対象に含めるセクションを選択できること
- コピー処理は本文を書き換えず、出力整形のみで完結すること

### 15.9 曲紐付け

- 生成曲の URL またはファイルを登録できること
- どの LyricVersion から生成したか記録できること
- 曲タイトル、評価メモ、スタイルメモ、プロンプトメモを保存できること
- 同一 LyricVersion に複数の SongArtifact を紐付けられること
- URL は参照情報として保持し、外部サイト埋め込みは行わないこと
- 未保存の Working Draft を直接 SongArtifact へ紐付けないこと
- Working Draft 状態から曲紐付けを行う場合は、保存して LyricVersion を作成するか、既存 LyricVersion を選ぶ導線を出すこと

### 15.10 Style 保持

- Style 候補を保持できること
- 検索、絞り込み、補助メモとして使えること
- Style を品質判定の根拠にしないこと
- MVP では Project ごとに 1 件の active StyleProfile を編集できること
- StyleProfile は Project 設定または編集画面内メタ情報パネルから編集できること
- StyleProfile の削除は論理削除で扱い、削除済みデータ管理画面から復元できること

### 15.11 プロジェクト・ブラウズ UI

- 左ペインにプロジェクト一覧と歌詞版ツリーを表示できること
- 編集中の歌詞版を切り替えられること
- 関連する断片、曲、メタ情報へ移動しやすいこと
- 編集中の対象が Working Draft であることを UI 上で識別できること

### 15.12 インポート

- テキストファイルから断片または歌詞本文を取り込めること
- コピー & ペーストによる手動取り込みを妨げないこと
- インポート時に文字コード判定に失敗した場合は、ユーザーへ再選択手段を提示できること
- MVP で必須とするファイル形式は `.txt` とすること
- CSV / Markdown の直接インポートは将来拡張とすること
- インポートは専用ダイアログまたは同等の明示導線から実行できること
- Windows / macOS では OS ネイティブのファイル選択ダイアログを利用すること
- PoC では `.txt` フィルタ付きの単一ファイル選択ダイアログを既定とすること

### 15.13 エラー表示

- 保存失敗、読み込み失敗、コピー失敗時にユーザーが原因を把握できるメッセージを表示すること
- エラー表示は創作作業を不必要に遮らず、再試行可能であること

### 15.14 設定

- コピー整形設定を保持できること
- モデル頻出表現チェックの実行パラメータを設定できること
- 既定セクションプリセットを設定から切り替えず、MVP では固定提供とすること
- アプリ設定はローカル保存されること
- PoC では少なくとも `llm_enabled`、`llm_base_url`、`llm_model_name`、`llm_timeout_ms`、`llm_max_output_tokens`、`llm_temperature` を設定できること
- `app_settings` はアプリ全体に効く設定のみを保持し、Project 固有データや LyricVersion 固有データは保持しないこと
- 最後に使ったコピー整形設定は `app_settings` に保存してよいこと
- LyricVersion 保存時に記録する copy 設定は、その時点の出力条件を履歴として保持するためのスナップショットであり、`app_settings` とは別責務とすること

### 15.15 削除済みデータ管理

- 論理削除された Project、LyricVersion、SongArtifact、CollectedFragment、RevisionNote、StyleProfile を確認できること
- 削除済みデータを種別ごとに復元できること
- MVP では物理削除 UI を通常導線へ露出しないこと
- 削除済みデータ一覧の既定表示は削除バッチ単位とすること
- バッチ内の個別対象を展開確認できること
- Project、LyricVersion の復元はバッチ単位で実行できること

### 15.16 エクスポート

- Project 単位で手動エクスポートを実行できること
- 出力先フォルダを指定できること
- 論理削除データを含めるかどうかを選択できること
- エクスポート完了後に生成ファイルの場所を確認できること
- 生成ファイルパスの表示だけで MVP の完了要件を満たすこと
- PoC では保存ダイアログ方式を採用し、既定拡張子を `.lyrlytic.zip` とすること

## 16. モデル頻出表現チェック要件

### 16.1 基本方針

モデル頻出表現チェックは、ローカル LLM による局所レビュー機能である。  
全文の常時監視や自動補正は行わない。

PoC では、ユーザー環境で起動済みの OpenAI 互換ローカル HTTP API へ接続する方式を既定とする。  
加えて `llama.cpp` に限り、LyricLytic から `llama-server` を直接起動し、その localhost API へ接続する補助モードを許容する。

### 16.2 発火条件

- ユーザーが明示的に実行した時のみ走ること
- 対象は歌詞全文とし、ユーザーが 1 フレーズ以上の確認対象テキストを明示入力した時のみ実行できること
- 設定でオン / オフ可能であること
- オフ時には一切動作しないこと
- 接続設定未完了または接続確認未成功時は実行できないこと

### 16.3 判定処理

- 入力フレーズと歌詞全文を文脈としてローカル LLM で N 回サンプリングを行うこと
- 生成結果中に繰り返し出現するフレーズ単位の表現を集計すること
- 各表現について出現割合を算出すること
- 一定割合以上の表現をモデル頻出表現として警告表示すること
- `凡庸` / `奇抜` の 2 方向を切り替えられること
- `凡庸` は閾値以上、`奇抜` は閾値以下を対象にできること
- 閾値未満の表現は結果一覧から除外すること
- 結果一覧は出現件数の降順で表示すること
- 実行条件で使う N、閾値、対象長さの既定値は設定ファイルまたはアプリ設定で変更可能とすること
- N は小さい試行回数に固定せず、PoC では 100 倍単位の大きめサンプル数を指定できること
- MVP の既定実行基盤は、ユーザー環境上のローカル LLM ランタイムを利用する方式を優先すること
- PoC では OpenAI 互換 Chat Completions API を既定接続方式とすること
- PoC では構造化 JSON を返させ、アプリ側でパースすること

### 16.4 意味づけ

本機能は、表現の良否や禁止を判定するものではない。  
あくまで、そのローカルモデル上で出やすい表現の偏りを可視化する機能である。

### 16.5 表示

少なくとも以下を確認できること。

- 該当表現
- 出現件数
- 出現割合
- 入力フレーズ
- 簡易理由または補助説明
- 候補フレーズのローマ字・母音・子音
- 入力フレーズとの韻の近さ

### 16.6 実行上限

- 通常編集を阻害しないため、同一セクションに対する連続実行にはキャンセルまたは上書き制御を持つこと
- 実行中であることを UI 上で識別できること
- PoC では接続先を `127.0.0.1` または `localhost` に限定すること
- `llama.cpp` 直起動モードでは、実行ファイルパスとモデルパスを設定できること

## 17. 低頻出候補レコメンド要件

### 17.1 基本方針

低頻出候補レコメンドは、モデル頻出表現チェックの結果を踏まえ、相対的に出現割合の低い候補表現を提示する局所リライト支援機能である。

### 17.2 要件

- モデル頻出表現チェックの結果に基づき、低頻出候補を提示できること
- 候補は該当箇所または該当セクションの見直し用として参照できること
- 候補はそのままコピーまたは挿入できてもよい
- 候補は強制置換ではなく、参考候補として扱うこと
- 候補を採用しても既存本文は自動置換しないこと
- 候補生成の実行元となったチェック結果との対応が分かること
- PoC ではモデル応答を JSON 構造として受け取り、候補一覧へ変換すること
- `凡庸` 方向ではより通俗的で出やすい候補、`奇抜` 方向ではより低頻出で意外性のある候補を提示できること

### 17.3 意味づけ

低頻出候補は「良い表現」を保証するものではない。  
モデル頻出表現とは異なる方向の別案を提示するための補助機能である。

## 18. AI 支援方針

### 18.1 全文生成を主目的としない

本プロダクト内で歌詞全文のドラフト生成を主機能としない。  
必要なドラフトは外部で生成し、本プロダクトへインポートして扱う。

### 18.2 補助の対象

AI 支援は当面、以下の局所支援に限定する。

- モデル頻出表現チェック
- 低頻出候補レコメンド
- 将来的な局所言い換え
- 指定セクションの限定的補助

### 18.3 モデル方針

- ローカルモデルを前提とする
- 通常モデルを基準にする
- 極端な改造モデルは標準採用しない
- 補助判定は参考情報として扱う
- MVP ではアプリ同梱モデルを必須前提にせず、既存ローカル実行環境との接続を優先する
- PoC では OpenAI 互換ローカル HTTP API 接続を 1 系統だけ実装対象とする
- `llama.cpp` は、OpenAI 互換 localhost API を保ったままアプリ側から起動・停止を管理できてもよい

### 18.4 学習データ方針

- 著作権上不明瞭な流行歌スクレイピングを前提にしない
- 自作歌詞、許諾済み歌詞、辞書型データ、ノウハウデータを優先する
- 学習機能を入れるとしても、小規模・ローカル・非配布前提とする

独立画面を増やしすぎず、周辺機能は可能な限り同一文脈内のパネル、ドロワー、ダイアログで扱う。

## 20. 技術方針

### 20.1 採用方針

想定技術スタックは以下とする。

- フロントエンド: React または Vue
- エディタ: Monaco Editor
- デスクトップ基盤: Tauri
- バックエンド: Rust
- 永続化: SQLite
- 初期サポート OS: Windows, macOS

### 20.2 採用理由

- Monaco Editor は VS Code 由来で編集体験と差分表示に優れる
- Tauri は OS 標準 WebView を利用し、Electron より軽量で権限制御しやすい
- SQLite はローカル完結型アプリと相性がよく、MVP に適したシンプルな永続化手段である

## 21. セキュリティ要件

Tauri を採用するにあたり、以下を要件として遵守する。

- 外部コンテンツロード禁止
- Remote API 禁止
- 任意埋め込み iframe 禁止
- Shell / External Launcher の利用を原則禁止し、必要時もホワイトリスト制御する
- Rust 側コマンド公開範囲を必要最小限に絞る
- capabilities を最小権限で構成する
- 開発用サーバーは安全が保証されたネットワークでのみ使用する

### 21.1 具体方針

- ローカルにバンドルされた資産のみ読み込む
- CSP は `default-src 'self';` を中心に構成する
- ファイルシステム権限はプロジェクトに必要な範囲へ限定する
- 外部通信機能は MVP で実装しない
- 外部 URL は保存・表示のみを基本とし、自動アクセスしない
- 未知の外部ページ内容をアプリ内でレンダリングしない
- 外部 URL を開く場合は、Windows / macOS ともにユーザーの明示操作時のみ既定ブラウザを使うこと
- PoC では外部 URL を開く前に常に確認ダイアログを出すこと
- PoC のローカル LLM 接続先は `127.0.0.1` または `localhost` のみに限定し、外部ホストを受け付けないこと

## 22. 非機能要件

### 22.1 操作性

- キーボード中心で快適に操作できること
- 編集中に補助 UI が邪魔しないこと
- 書いている最中の集中を壊さないこと

### 22.2 応答性

- テキスト入力に知覚可能な遅延がないこと
- コピーは即時であること
- 検索は短時間で返ること
- モデル頻出表現チェックは通常編集を阻害しないこと
- 通常の本文編集操作は 100ms 未満で応答し、入力詰まりを感じにくいこと
- プロジェクト切替と履歴一覧表示は 1 秒以内を目標とすること
- 全文コピーは 1 秒以内に完了すること
- PoC では Windows / macOS の両方でインポート、エクスポート、クリップボードコピーが主要フローとして通ること

### 22.3 可逆性

- 誤操作しても自動保存状態へ復帰できること
- スナップショットから過去版へ戻れること
- どの版から何を採用したか追いやすいこと

### 22.4 安全性

- 完全ローカルで動作すること
- 外部通信を行わないこと
- 権限は最小限とすること
- 未信頼外部コードを読み込まないこと
- 外部 URL を保存しても自動取得しないこと
- ローカルファイルパスは参照情報であり、OS 間可搬性を保証しないこと

### 22.5 保守性

- ドメインモデルと UI 表示モデルの責務を分離しやすい構成であること
- 設定値、閾値、プリセットは将来変更しやすい形で定義すること

## 23. 開発上の主なリスク

- Monaco Diff Editor と履歴 UI の整合設計が複雑になりやすい
- セクション構造と本文編集の両立で UI / データモデルが肥大化しやすい
- バージョン管理ロジックで親子関係や復元フローの整合性が崩れやすい
- 補助判定機能が重い場合、編集体験を阻害する可能性がある
- Tauri 設定ミスによる権限過剰付与や CSP 漏れのリスクがある

## 24. 受け入れ基準

以下を満たした場合、本要件定義に基づく MVP を満たしたものとする。

- プロジェクトを作成し、歌詞を編集できる
- 新規 Project 作成直後は空の Working Draft から開始できる
- セクションをプリセットから追加し、任意名称でも追加できる
- 同名追加時に連番が付与される
- 編集中の状態が自動保存される
- 保存ボタンでスナップショット版が作成される
- 過去版と差分比較できる
- 推敲メモを版 + セクションに紐付けられる
- 断片を管理し、本文へ流し込める
- 検索パネルから `本文 / 過去版 / 断片 / タグ` を切り替えて検索できる
- 全文 / セクション単位で整形コピーできる
- 曲を歌詞版に紐付けられる
- StyleProfile を Project 単位で保持、編集、論理削除 / 復元できる
- Project 単位エクスポートを実行できる
- 論理削除した主要データを削除済みデータ管理画面から復元できる
- モデル頻出表現チェックを手動で実行できる
- 頻出表現の割合を表示できる
- 低頻出候補を参考候補として提示できる
- すべてがローカル環境内で完結する

### 24.1 受け入れテスト観点

- 新規 Project 作成から歌詞編集開始までが正常に行えること
- アプリ再起動後に Working Draft が復元されること
- 同一 Project で複数 LyricVersion を保存し、2 版比較できること
- 同一 LyricVersion へ複数 SongArtifact を紐付けられること
- テキストファイルから断片を取り込み、本文へ挿入できること
- 破壊的操作時に確認導線が出ること
- 外部通信なしで主要フローが成立すること
- 論理削除したデータを復元できること

### 24.2 詳細テストケース

詳細な受け入れ手順と期待結果は `acceptance-test-cases-v1.md` を正本とする。  
MVP / PoC 判定時は、本節の観点に加えて個別テストケースの合否を確認すること。

## 25. 実装上の優先原則

実装時は以下の原則を優先する。

1. 創作の邪魔をしないこと
2. 自由な編集を殺さないこと
3. 自動保存と履歴保存を混同しないこと
4. 表示名と内部識別子を分離すること
5. 判定系機能を真理値として扱わないこと
6. ローカル完結方針を崩さないこと

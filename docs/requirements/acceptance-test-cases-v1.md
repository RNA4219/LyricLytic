# LyricLytic 受け入れテストケース v1

## 1. 目的

本書は、`requirements.md` の受け入れ基準を、実際に確認可能なテストケース粒度へ落とし込むための文書である。  
PoC / MVP のレビュー時に「何をどの順で確認し、何をもって合格とするか」を揃えることを目的とする。

## 2. 前提

- 対象 OS は Windows 11 と macOS 最新安定版
- 1 ユーザー、1 ローカル端末利用を前提とする
- テストデータはローカルに作成し、クラウド同期は使わない
- LLM 関連ケースはローカルホスト上の OpenAI 互換 API が起動済みの場合のみ実施する

## 3. 判定ルール

- `必須`: MVP / PoC 完了判断に必須
- `条件付き`: 設定やローカル LLM 接続など、実施条件が揃った時に確認
- `将来`: 現行スコープ外

不合格ケースが 1 件でもある場合、当該機能領域は未完了として扱う。

## 4. テストケース一覧

### TC-01 新規 Project 作成

- 区分: 必須
- 対象要件: Project 管理、ホーム画面、空 Draft 開始
- 手順:
  1. アプリを起動する
  2. ホーム / プロジェクト選択画面で新規 Project を作成する
- 期待結果:
  - 歌詞編集画面へ遷移する
  - 新規 Project 作成直後は空の Working Draft で開始する
  - セクションプリセットは自動挿入されず、追加候補として認識できる

### TC-02 セクション追加

- 区分: 必須
- 対象要件: セクション追加、命名規則
- 手順:
  1. 空 Draft に `Verse` を追加する
  2. さらに同名の `Verse` を追加する
  3. 任意名称のセクションを追加する
- 期待結果:
  - プリセットセクションを追加できる
  - 同名追加時は連番が付与される
  - 任意名称のセクションを追加できる

### TC-03 Working Draft 自動保存

- 区分: 必須
- 対象要件: 自動保存、Working Draft
- 手順:
  1. Section 本文を編集する
  2. 一定時間待機する
  3. アプリを閉じて再起動する
- 期待結果:
  - 編集内容が Working Draft に保存される
  - 再起動後に直前の作業状態を復元できる
  - 履歴版は自動では増えない

### TC-04 スナップショット保存

- 区分: 必須
- 対象要件: スナップショット保存、履歴管理
- 手順:
  1. Working Draft を編集する
  2. 保存ダイアログを開く
  3. `snapshotName` と任意メモを入力して保存する
- 期待結果:
  - LyricVersion が新規作成される
  - 保存後も編集対象は Working Draft のままである
  - 履歴一覧から保存版を識別できる

### TC-05 差分比較

- 区分: 必須
- 対象要件: 差分比較
- 手順:
  1. 同一 Project で 2 件以上の LyricVersion を保存する
  2. 差分確認画面または同等 UI から 2 版を選択する
- 期待結果:
  - 2 版の差分を表示できる
  - 少なくとも行単位の差分が分かる
  - 比較中に元データを破壊しない

### TC-06 過去版からの再開

- 区分: 必須
- 対象要件: 復元、Working Draft 再構築
- 手順:
  1. 過去の LyricVersion を選ぶ
  2. `この版から作業を再開` を実行する
- 期待結果:
  - 既存 LyricVersion は更新されない
  - Working Draft が選択版の内容で再構築される
  - 再構築後に通常編集を継続できる

### TC-07 RevisionNote 登録

- 区分: 必須
- 対象要件: 推敲メモ
- 手順:
  1. 保存済み LyricVersion の任意 Section を選ぶ
  2. 推敲メモを追加する
- 期待結果:
  - 版 + セクションに紐付いて保存される
  - 必要に応じてメモ種別を設定できる
  - 他版へ自動継承されない

### TC-08 断片の手動登録

- 区分: 必須
- 対象要件: 断片管理
- 手順:
  1. CollectedFragment を手動入力で登録する
  2. 一覧から対象断片を選ぶ
  3. 本文またはセクションへ挿入する
- 期待結果:
  - 断片を登録できる
  - 断片を本文へ反映できる
  - 必要に応じて `used` へ状態変更できる

### TC-09 `.txt` インポート

- 区分: 必須
- 対象要件: インポート
- 手順:
  1. `.txt` ファイルを選択する
  2. 断片取り込みまたは本文取り込みを選択する
- 期待結果:
  - `.txt` インポートが成功する
  - 断片または本文のどちらかへ反映できる
  - 文字コード失敗時は再選択導線が出る

### TC-10 コピー整形

- 区分: 必須
- 対象要件: コピー
- 手順:
  1. 全文コピーを実行する
  2. セクション単位コピーを実行する
  3. 見出し有無と空行保持設定を切り替える
- 期待結果:
  - 全文 / セクション単位のコピーができる
  - 出力整形のみが変化し、本文自体は変わらない
  - 最後に使った整形設定を保持できる

### TC-11 SongArtifact 紐付け

- 区分: 必須
- 対象要件: 曲紐付け
- 手順:
  1. 保存済み LyricVersion を選択する
  2. URL またはローカルファイルを登録する
  3. 曲タイトル、評価メモ、プロンプトメモを入力する
- 期待結果:
  - SongArtifact を保存できる
  - 同一 LyricVersion に複数件紐付けられる
  - Working Draft は直接紐付け対象に出ない

### TC-12 外部 URL オープン確認

- 区分: 必須
- 対象要件: セキュリティ、外部 URL
- 手順:
  1. 登録済み URL を開く操作を行う
- 期待結果:
  - 自動では開かれない
  - 確認ダイアログが表示される
  - キャンセル時はブラウザ起動しない

### TC-13 論理削除

- 区分: 必須
- 対象要件: 論理削除、削除済みデータ管理
- 手順:
  1. Project または LyricVersion を削除する
  2. 削除済みデータ管理画面を開く
- 期待結果:
  - 通常一覧から対象が消える
  - 削除済みデータ管理画面で対象を確認できる
  - 物理削除 UI は通常導線に露出しない

### TC-14 論理削除からの復元

- 区分: 必須
- 対象要件: 復元
- 手順:
  1. 削除済みデータ管理画面で削除対象を選ぶ
  2. 復元を実行する
- 期待結果:
  - バッチ単位で復元できる
  - 復元後に通常一覧へ戻る
  - 復元不可条件がある場合は理由を表示できる

### TC-15 StyleProfile 編集

- 区分: 必須
- 対象要件: Style 保持
- 手順:
  1. Project の StyleProfile を作成または編集する
  2. tone、tags、memo などを更新する
- 期待結果:
  - Project 配下で active 1 件の StyleProfile を保持できる
  - 検索 / 絞り込み / 補助メモ用途として参照できる
  - 品質判定には使われない

### TC-16 StyleProfile 論理削除と復元

- 区分: 必須
- 対象要件: StyleProfile 削除 / 復元
- 手順:
  1. StyleProfile を削除する
  2. 削除済みデータ管理画面から復元する
- 期待結果:
  - StyleProfile は論理削除される
  - 削除済みデータ管理画面に表示される
  - 復元後に再び active として扱える

### TC-17 検索パネル

- 区分: 必須
- 対象要件: 検索 UI
- 手順:
  1. 歌詞編集画面で検索 UI を開く
  2. `本文 / 過去版 / 断片 / タグ` を切り替えて検索する
- 期待結果:
  - 専用画面に遷移せず、編集コンテキストを維持したまま検索できる
  - 種別ごとに対象データへ移動できる
  - 過去版横断検索に RevisionNote は含まれない

### TC-18 エクスポート

- 区分: 必須
- 対象要件: エクスポート
- 手順:
  1. Project 単位エクスポートを実行する
  2. 出力先を指定する
  3. 必要に応じて論理削除データ含有オプションを切り替える
- 期待結果:
  - `.lyrlytic.zip` を生成できる
  - `app_settings` は含まれない
  - 生成ファイルパスを確認できる

### TC-19 ローカル完結

- 区分: 必須
- 対象要件: ローカル完結
- 手順:
  1. ネットワーク切断環境で主要フローを実施する
- 期待結果:
  - Project 作成、編集、保存、差分、断片、削除復元、エクスポートが成立する
  - 外部通信必須の前提がない

### TC-20 モデル頻出表現チェック

- 区分: 条件付き
- 対象要件: モデル頻出表現チェック
- 手順:
  1. ローカル LLM 接続設定を行う
  2. 接続確認を成功させる
  3. 任意のフレーズを入力する
  4. 歌詞全文が入った状態で頻出表現チェックを実行する
- 期待結果:
  - 明示実行時のみ動作する
  - 入力フレーズを起点として歌詞全文を文脈に使う
  - 出現件数と出現割合付きで結果を表示できる
  - 閾値未満の結果は一覧に出ない
  - 結果が出現件数の降順で並ぶ
  - `凡庸` / `奇抜` の切替で抽出方向を変更できる
  - 候補にローマ字 / 母音 / 子音と韻の近さを表示できる
  - 未接続時は実行できない

### TC-21 低頻出候補レコメンド

- 区分: 条件付き
- 対象要件: 低頻出候補レコメンド
- 手順:
  1. 頻出表現チェック結果を得る
  2. 低頻出候補レコメンドを実行する
- 期待結果:
  - 候補一覧を表示できる
  - 強制置換はされない
  - 候補生成元のチェック結果との対応が分かる

### TC-22 OS 別主要フロー確認

- 区分: 必須
- 対象要件: Windows / macOS 差分
- 手順:
  1. Windows 11 で主要フローを通す
  2. macOS 最新安定版で主要フローを通す
- 期待結果:
  - `.txt` インポート、コピー、保存、差分、削除復元、エクスポートが両 OS で成立する
  - ショートカット表記が OS に応じて切り替わる
  - URL オープン前の確認挙動が両 OS で成立する

## 5. 最小合格ライン

MVP / PoC の最小合格ラインは以下とする。

- `TC-01` から `TC-19` と `TC-22` がすべて合格
- `TC-20` と `TC-21` は、ローカル LLM 接続を対象範囲に含めるリリースでは合格必須

## 6. 要件への戻し方

以下が見つかった場合は、実装で局所吸収せず要件へ戻す。

- テストケースが現行要件から再現できない
- 同一テストケースで OS ごとに合格条件が変わる
- スキーマや UI フローの前提不足で、期待結果を一意に判定できない
